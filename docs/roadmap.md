# WAVEx Implementation Roadmap

> Status: concrete working roadmap. This is intentionally implementation-facing and should be updated as slices land.

## Current baseline

The active implementation is this repository, a TypeScript/pnpm monorepo.

Implemented enough to build demos:

- `@wavex/core`
  - Parses `.wx` files split by the required `~~~` separator (absorbed the former `@wavex/language-core`), plus the shared project model: config, route/resource naming, utility helpers.
  - Supports strict indentation, native elements, `@` components, `+head`, `+if`, `+for`, Convex references/calls, colon attributes, semantic events, utility groups, inline text, expression lines, and the MVP prose spans.
  - The single source of truth for `.wx` parsing, consumed by the compiler and the future `@wavex/lsp`.
- `@wavex/compiler`
  - Emits Lit render modules from `.wx` templates.
  - Exports `resources` metadata for `$$module:function` declarations.
  - Emits semantic event attributes as `data-wx-*` markers.
- `@wavex/runtime` (Lit renderer at `@wavex/runtime/lit`, absorbed the former `@wavex/renderer-lit`)
  - Provide render context creation, Lit mounting, rerendering, render replacement for HMR, and delegated semantic events.
  - Include a tested generic resource controller, resource lifecycle state, a Convex browser-client adapter, semantic mutation/action dispatch, pending/error action state, and a Lit mount path that can consume page-module `resources` metadata.
- `@wavex/vite-plugin`
  - Transforms `.wx` modules in Vite/Vite+.
  - Discovers page routes for `virtual:wavex/routes`.
  - Provides `/@wavex/bootstrap` as the default app entry, mounting `src/pages/index.wx`, importing `src/style.css`, and wiring Convex/resource/action clients.
  - Discovers local components and handles basic `.wx` HMR.
  - Uses only `src/pages` / `src/components` for page and component discovery.
- `wavex` CLI
  - Provides `check`, `routes`, and `compile` commands, with `dev` / `build` proxying to Vite for now.
- Demo app
  - `apps/todo/` builds as a Convex-backed single-page `.wx` app with query resources, semantic form/click mutations, keyed loops, and HMR wiring.

Current known gap: manual browser/dev-server verification of the Convex-backed TODO flow is still needed; automated Convex function tests and production build pass.

## MVP north star

A small WAVEx TODO app should be authored primarily in `.wx` and Convex files:

- `$$tasks:list` declares a realtime task resource.
- Template reads tasks through WAVEx-provided resource context.
- Native forms/buttons invoke Convex mutations via semantic events such as `:submit:$$tasks:create`.
- The default app entry is generated by WAVEx bootstrap; an app-authored `src/main.ts` is not required for the standard Convex-backed flow.
- `pnpm check` and `cd apps/todo && pnpm build` pass.

## Ordered implementation slices

### 1. Convex resource bridge

Status: runtime/compiler foundation landed; live TODO app wiring remains part of the real Convex conversion slice.

Goal: make compiled `resources` metadata do useful runtime work.

Tasks:

- Add a runtime resource abstraction that can be tested without a live Convex deployment.
- Add a Convex-backed implementation using the official Convex browser client.
- Teach the mount path to accept a page module's `resources` export alongside `render`.
- Subscribe to declared `$$module:function` query resources.
- Store resource values in `context.resources[name]`.
- Track minimal resource status for loading and error states.
- Rerender when resource values or statuses change.

Done criteria:

- Unit tests cover subscription lifecycle, update-triggered rerender, and cleanup.
- A fake resource client can drive `context.resources.tasks` in tests.
- Existing compiler/parser tests still pass.
- `pnpm check` passes.

### 2. Mutation and form bridge

Status: runtime foundation landed; live TODO app wiring remains part of the real Convex conversion slice.

Goal: make semantic events target Convex mutations/actions directly.

Tasks:

- Resolve semantic event targets that look like `$$module:function`.
- Add a runtime mutation/action invocation path.
- Convert native form submissions into MVP mutation args.
- Support element-derived args for buttons/inputs, starting with explicit `data-*` attributes.
- Track minimal pending/error state for mutation targets.
- Keep custom handler dispatch working for non-Convex targets such as `:click:openMenu`.

Done criteria:

- Unit tests cover submit, click, pending, error, and custom-handler dispatch behavior.
- Form submission calls the configured mutation client and prevents default navigation.
- `pnpm check` passes.

### 3. Convert the TODO demo to real Convex

Status: initial conversion landed with Convex schema/functions, convex-test coverage, resource-driven `.wx`, and generated WAVEx bootstrap replacing app-authored mount glue.

Goal: prove the primary WAVEx promise in the demo app.

Tasks:

- Implement `apps/todo/convex/schema.ts` and `apps/todo/convex/tasks.ts` with list/create/toggle/delete operations.
- Rewrite `apps/todo/src/pages/index.wx` to declare and render `$$tasks:list`.
- Replace localStorage state in `apps/todo/src/main.ts` with thin WAVEx mount + Convex configuration.
- Use semantic mutation events for add, toggle, delete, and clear-completed behavior.
- Keep the UI small and focused; do not add routing or layout work in this slice.

Done criteria:

- `pnpm check` passes.
- `cd apps/todo && pnpm build` passes.
- The TODO app can list, create, toggle, and delete Convex tasks in dev.
- Manual `src/pages/index.wx` edits still HMR without losing the Convex client.

### 4. Web Awesome adoption in the TODO app

Status: not started. The current demo uses native elements with bespoke classes (`todo-shell`, `icon-button`, `empty-list`, …) and ~200 lines of custom CSS; the Web Awesome Pro and Font Awesome kit packages are installed in `apps/todo/package.json` but unused.

Goal: make the TODO app the first real proof of the first-class Web Awesome / Font Awesome promise before multi-route work starts.

Tasks:

- Replace bespoke widgets with Web Awesome components via `@` shorthand (`@input`, `@button`, `@checkbox`, `@card`, `@callout`), starting from `.agents/skills/using-webawesome/patterns/` per AGENTS.md rather than inventing markup.
- Replace custom CSS with Web Awesome utility classes and design tokens; shrink `src/style.css` to app-specific theming only.
- Use `@icon` for the add and delete affordances instead of text glyphs (the current `×` button).
- Resolve the utility-shorthand divergence: the spec writes `[stack gap:xl]` while the design doc, parser, and demo use `[stack gap-xl]`; pick one form and synchronize spec, design doc, parser tests, and the demo.
- Adopt the spec's resource-state directives (`+loading`, `+empty`) in place of `+if !tasks` checks, adding parser/compiler/runtime support if not yet landed.

Done criteria:

- The TODO app renders with `<wa-*>` components, utilities, and `@icon`; bespoke CSS is reduced to theming.
- Spec, design doc, and tests agree on the utility shorthand form.
- `pnpm check` and `cd apps/todo && pnpm build` pass.

### 5. Multi-route runtime

Goal: turn discovered routes into client navigation, not just metadata.

Tasks:

- Generate or expose enough route module information for runtime loading.
- Add a client router using History API and native `a href` interception; evaluate `@lit-labs/router` before hand-rolling matching/loading, per the design doc's reuse principle.
- Populate `context.route.path`, `context.route.params`, and `context.route.query`.
- Preserve current resource/client state across same-app navigation.
- Keep route matching aligned with `createRouteDefinition` semantics.

Done criteria:

- Multi-page fixture covers static, dynamic, and catch-all routes.
- `virtual:wavex/routes` supports the runtime needs.
- `wavex routes` output and runtime route matching agree.
- `pnpm check` passes.

### 6. Template components and layouts

Goal: make reusable `.wx` components and route layouts credible.

Tasks:

- Define the runtime/component contract for local `.wx` components.
- Import or register local components discovered from `src/components`.
- Implement `+layout.wx` composition through native `slot` semantics.
- Keep `@wa/...` / Web Awesome shorthand separate from local component resolution.
- Add fixture/demo coverage for nested layouts and reusable components.

Done criteria:

- Local components render from `src/components`.
- `+layout.wx` composes sibling and descendant pages.
- `apps/todo/` remains compatible with the fixed `src/pages` convention.
- `pnpm check` and `cd apps/todo && pnpm build` pass.

### 7. Head management and prerender baseline

Goal: make `+head` useful beyond compiled render output.

Tasks:

- Add a runtime head manager for `title`, `meta`, and `link`.
- Connect route changes and resource updates to head updates.
- Add the smallest credible prerender path for static route HTML; evaluate `@lit-labs/ssr` since compiled output is Lit templates.
- Keep prerendering an output optimization, not a server-runtime dependency.

Done criteria:

- `+head` updates browser document head in dev.
- Build/prerender emits route HTML for simple static pages.
- Existing client-only behavior still works.

### 8. Web Awesome and Font Awesome capability detection

Goal: move from tag naming conventions to package-aware imports and diagnostics.

Tasks:

- Detect installed Web Awesome free/pro packages without bundling licensed assets.
- Detect installed Font Awesome packages, kits, or local icon assets.
- Generate imports/assets only for used capabilities.
- Surface missing capability diagnostics in `wavex check`.

Done criteria:

- Free package setup works by default.
- Pro capabilities are unlocked only when installed by the app owner.
- Missing component/icon references produce actionable diagnostics.

### 9. Error boundaries and PostHog

Goal: cover app reliability and first-class analytics after the core app loop works.

Tasks:

- Implement local `+boundary` and route `+error.wx` behavior.
- Add pageview capture for route transitions.
- Add custom event capture for selected semantic actions.
- Decide official PostHog client versus minimal capture bridge for the baseline.

Done criteria:

- Route/render/resource errors show deterministic error UI.
- Pageview capture works in a demo app.
- Analytics remains optional/configured, not required for local development.

### 10. Volar LSP baseline — may start in parallel with slices 5–9

Status: not started. Decision (2026-06): the `@wavex/core` TS parser is the single source of truth for `.wx`; the earlier Tree-sitter grammar and Zed dev extension were deleted rather than maintained as a second grammar definition. Editor support is LSP-first via Volar, which works across editors (VS Code, Zed, Neovim, Helix) without per-editor grammars.

Goal: deliver the typed-template DX (template expressions checked against prelude/Convex/Web Awesome types — the main reason to prefer `.wx` over TSX), built directly on the `@wavex/core` parser. This slice depends only on the parser, which is already the stable single source of truth, so it is numbered last but explicitly **not** gated on slices 5–9; start it as soon as there is appetite.

Tasks:

- Stand up `@wavex/lsp` on Volar over the `@wavex/core` AST (which already carries source ranges).
- Map `.wx` preludes and template expressions to virtual TypeScript documents for diagnostics, hover, and completions.
- Emit semantic tokens from the parser for editor highlighting.
- Start LSP diagnostics/completions for components, Convex functions, routes, and attributes.
- Optionally add a thin TextMate grammar for highlight-before-LSP-boots and GitHub views (drift-tolerant, cosmetic only).

Done criteria:

- Editors highlight `.wx` files via LSP semantic tokens.
- LSP can report core parser diagnostics and offer MVP completions.
- Template expressions type-check against prelude/Convex types in at least one proven case.
- No second grammar definition exists; the LSP consumes the same parser as the compiler.

## Post-TODO validation app

Once the TODO app has exhausted what it can highlight (including the slice 4 Web Awesome adoption), the full-feature validation target is the Swell Conf companion app planned in `docs/validation-app.md`. It is the proving ground for slices 5–9; build it incrementally as those slices land.

## Explicit deferrals

Do not start these before the Convex-backed TODO milestone unless a slice explicitly requires them:

- Editor work beyond the Volar LSP baseline (per-editor extensions, TextMate polish). The Volar LSP baseline itself is slice 10 and may run in parallel with slices 5–9.
- Prerender/SSR beyond the smallest head/static route baseline.
- Web Awesome/Font Awesome capability system before the Convex runtime loop is proven.
- Multi-route expansion before a single-page Convex app works.
- PostHog before routing/resources/mutations are stable enough to instrument.
- Zig/WASM browser runtime work.

## Working rules

- Keep slices small and testable.
- Prefer runtime abstractions that can be tested with fake clients before requiring a live Convex deployment.
- Keep `.wx` syntax changes tied to user-facing examples and tests.
- After each implementation slice, update this roadmap if the next slice or done criteria changes.
